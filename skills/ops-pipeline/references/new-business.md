# 新业务接入规范

当遇到不在现有业务规范中的新业务线时，按以下步骤确认配置，
然后用通用模式处理，并考虑是否需要补充业务规范文件。

## 需要确认的信息

向用户或从已有计划中确认以下内容：

1. **Coding 项目名**：`list_projects` 查找
2. **代码仓库**：`list_depots` 查找，确认 depot_id
3. **是否有独立 K8s 配置仓库**
   - 有 → 使用模式 B，需要确认配置仓库地址和 yaml 路径规则
   - 没有 → 使用模式 A，Dockerfile 在代码根目录
4. **目标环境**：test / prod，对应不同的 TKE_NS 和凭据
5. **镜像 namespace**：从已有同类计划推断（如 `cyouops/`、`web-wbhd/`）
6. **凭据 ID**：从已有同类计划的 `describe_ci_job` 获取

## 模式选择决策

```
代码仓库有 dockerfile/ 目录？
├─ 是 → 模式 B（双仓库），K8s yaml 由独立配置仓库管理
└─ 否 → 代码根目录有 Dockerfile？
         ├─ 是 → 模式 A（单仓库）
         └─ 否 → 需要用户提供 Dockerfile 位置
```

## 补充业务规范

如果该业务线会持续新增服务，建议在 `references/` 下创建对应规范文件：

```markdown
# {业务线名称} 业务规范
## 业务特征
## 适用 Pipeline 模式
## 环境变量规范
## 凭据 ID 获取方式
## 命名规范示例
```
